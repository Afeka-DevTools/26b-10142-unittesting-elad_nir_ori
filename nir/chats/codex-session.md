## Unit-testing explanation

### Student prompt

> I noticed that `App.java` does not contain a `main` method and instead
> provides a collection of utility functions. Please explain how JUnit
> discovers and runs the tests, how assertions determine whether a test passes,
> and how each function is tested.

### Codex response

Codex explained that each method marked with `@Test` is discovered and executed
by JUnit. A test calls one unit of the application and then compares its actual
behavior with the expected behavior:

```java
@Test
void exampleAdditionTest() {
    int result = App.add(3, 4);
    assertEquals(7, result);
}
```

If the expected and actual values match, the test passes. If they differ, or if
an unexpected exception occurs, JUnit reports a failure. Exception behavior is
tested explicitly with `assertThrows`, and lists are compared with
`assertIterableEquals`.

## Review of test sufficiency

The suite covers all public methods and the important documented branches. It
includes normal behavior, boundaries, empty values where supported, and
documented invalid input.

The tests are sufficient for the current written requirements, but a finite
test suite cannot prove that a program contains no defects. Additional expected
behavior would need to be defined before testing cases such as:

- `null` arguments
- integer overflow in `add`, `factorial`, or Fibonacci generation
- an empty string passed to `mostCommonWord`
- ties between equally frequent words
- locale-specific or normalized Unicode comparisons
